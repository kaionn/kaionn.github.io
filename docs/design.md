# ポートフォリオサイト トップページ構成案

対象: 転職(メイン)+ フリーランス営業(サブ)
掲載プロジェクト: signal-lab / tech-news-daily
実装想定: Next.js 静的エクスポート + Tailwind + GitHub Pages

公開先: `kaionn/kaionn.github.io`（ユーザーページ。**basePath 不要**）

---

## サイト全体のコンセプト

**「動き続ける仕組みを作るエンジニア」**

signal-lab も tech-news-daily も、共通点は「一度作ったら自動で回り続けるパイプライン」であること。アプリを並べるのではなく、この一点で自己紹介からプロジェクト紹介まで貫く。

転職側には「設計・自動化・運用まで一人で見られる」、フリーランス側には「放置されたシステムを"回り続ける状態"に戻せる」というメッセージとして両取りできる。

### シグネチャ要素: サイト自体が「動いている証拠」になる

ヒーロー直下に、GitHub API から取得したライブステータスを表示する。

- tech-news-daily の最終更新日時(= 今朝も自動配信された証拠)
- signal-lab の直近の Probe 数 / 判定結果

ポートフォリオの主張(動き続ける仕組み)を、サイト自身がリアルタイムに証明する構造。これが他の人のポートフォリオと差がつく一点。静的エクスポートでもクライアントサイド fetch(`api.github.com/repos/.../commits?per_page=1` など、未認証60req/hで十分)で実現可能。取得失敗時は行ごと非表示にしてレイアウトが崩れないようにする。

**データソース（確認済み）:**

- tech-news-daily 最終更新: `GET https://api.github.com/repos/kaionn/tech-news-daily/commits?per_page=1` → `[0].commit.committer.date`
- signal-lab Probe 数: `GET https://api.github.com/repos/kaionn/signal-lab/contents/experiments` → `type == "dir"` かつ名前が `_` で始まらないものを数える

- CTA: 「Projects を見る」(ページ内アンカー)/ GitHub アイコンリンク

**コピー ドラフト:**

> Kaion — Full-stack Engineer
>
> 作って終わりにしない。
> 設計からインフラ、CI/CD まで — 動き続ける仕組みごと届けるエンジニアです。

**ライブステータス行(例):**

> ● tech-news-daily — 今朝 06:00 に自動配信済み
> ● signal-lab — 現在 N 本の Probe が検証中

### 2. About(経歴サマリ)

3〜4行で本業の軸を示す。プロジェクトが個人開発2本なので、ここで業務スタックの重み(Go / AWS / Terraform)をしっかり出してバランスを取る。

**コピー ドラフト:**

> フルスタックエンジニア。業務では Rails / Go によるバックエンド開発、React / Next.js のフロントエンド、Terraform + GitHub Actions による AWS インフラの構築・運用まで一貫して担当。
> 個人では「小さく作って市場に聞く」検証ファースト開発を実践し、アイデア収集から公開・撤退判定までを自動化した仕組みを運用している。

### 3. Projects(2本、それぞれ深く)

各プロジェクトは同じ型で構成する: **課題 → 仕組み(図)→ 技術的な判断 → リンク**

数で勝負しない分、1本ごとの解像度を上げる。README の内容をそのまま貼らず、「なぜこう設計したか」に寄せて要約し直す。

#### 3-1. signal-lab

**見出し:** signal-lab — 検証ファースト開発の実験機構

**課題(ドラフト):**

> 旧機構ではアイデアから MVP を自動生成していたが、84案を生成して公開に至ったのは1本のみ。原因は「人間の判断がデータの前に置かれ、実験単価が高すぎた」こと。

**仕組み:**

> Issue 起票 → Claude Code が LP を生成して PR → merge で Vercel に自動公開 → 登録シグナルを Discord に即時通知 → 週次ワークフローが KILL / WATCH / GRADUATE を自動判定。
> 判断コストを人間からデータに移し、実験単価を「PR 1本分」まで下げた。

パイプライン図は README の ASCII 図をベースに、シンプルな横フロー図として再構成する(SVG or CSS)。サイト上で一番情報量のあるビジュアルはここに置く。

**技術的な判断(ドラフト、2〜3点):**

> - 撤退基準(verdict-rules.yaml)を人間の感覚ではなく設定ファイルに固定し、週次で機械的に判定させた
> - Probe を「動くマイクロツール」と「LP + 待機リスト」の2型に分け、案の規模で使い分けた
> - GitHub Actions を状態機械として使い、Issue / PR / デプロイを接続した

**リンク:** GitHub (https://github.com/kaionn/signal-lab) / 公開URL (https://signal-lab-six.vercel.app)

#### 3-2. tech-news-daily

**見出し:** tech-news-daily — 毎朝届く自動生成ニュースダイジェスト

**課題(ドラフト):**

> 技術ニュースの収集は毎日発生する定型作業。既存サービスの「量が多すぎる/興味領域と合わない」問題を、自分専用の自動パイプラインで解決した。

**仕組み:**

> 毎朝、Claude Code のルーチンが WebSearch でニュースを収集し、HTML を生成して GitHub Pages に自動デプロイ。前日分はアーカイブへ退避。人の手を介さず、公開から履歴管理までが毎日完結する。

**技術的な判断(ドラフト):**

> - 配信基盤を持たず「静的 HTML + Pages」に寄せ、運用コストをゼロに近づけた
> - アーカイブ退避を仕組みに含め、蓄積が壊れない構造にした

**リンク:** GitHub (https://github.com/kaionn/tech-news-daily) / 公開URL (https://kaionn.github.io/tech-news-daily)
※ ライブステータス行と連動していることに触れると効果的

## デザイン方針

> 2026-07-09: 見た目を「計器盤ダーク(コントロールルーム風)」へ刷新。テーマ「動き続ける仕組み」を、明るい紙面ではなく濃紺のダッシュボードで表現する。以下は刷新後の記述。旧・明るい紙面案の記述は削除済み。

テーマが「動き続ける仕組み」なので、計器・ログ・ステータス表示の語彙をデザインに借りる。全面ターミナル風(黒背景+蛍光緑のハッカー風)は既視感が強く上品さを欠くため避け、**濃紺のコントロールルームにパネル(枠付きカード)を並べ、ライブステータスを計器として主役に据える**構成にする。

### カラートークン

| 名前 | 値 | 用途 |
|---|---|---|
| bg | #0B1220 | ページ背景(濃紺) |
| panel | #101A2E | パネル面(bg よりわずかに明るい) |
| text | #E6EAF2 | 本文・見出し |
| muted | #8A94A6 | 補助テキスト・ラベル |
| signal | #2DD4BF | アクセント(ティール): リンク、CTA |
| status | #FBBF24 | ライブステータスのドット・強調(アンバー) |
| line | #223047 | パネル枠・罫線 |

### タイポグラフィ

- **英字見出し:** Space Grotesk(名前・セクション見出し)
- **本文:** Noto Sans JP
- **ステータス・タイムスタンプ・図中ラベル・パネルヘッダーラベル:** IBM Plex Mono

モノスペースは「機械が動いている」箇所(ライブステータス、パイプライン図、日時表示、パネルのヘッダーラベル・セクション番号)にだけ使い、それ以外は静かに保つ。使い分け自体が情報になる。

### レイアウト

- 1カラム、max-width 960px の計器盤レイアウト(読み物幅 720px から拡張)
- 各セクションは「パネル」(`border border-line bg-panel rounded-lg`、角丸は控えめ)としてコンテンツを収める。パネル上部に mono の小さなヘッダーラベル(例: `01 / ABOUT`、`02 / PROJECTS`。セクション番号は signal 色、セクション名は muted 色)を置く
- セクション間の罫線区切りは廃止し、パネルの並び(gap)で区切る
- ヒーローはパネル外(生の bg 上)に名前 + コピーを置き、直下の LIVE パネル(ヘッダー「LIVE」+ アンバーの pulse ドット)だけを独立した枠付きパネルとして目立たせる
- 背景に極めて薄いドットグリッド(text 色 4% 程度の radial-gradient)を敷き、うるさくしない範囲で計器盤の質感を出す
- モーション: ライブステータスのドットに控えめな pulse のみ。それ以外のスクロールアニメーションやパネルのホバーエフェクトは入れない
- レスポンシブ必須(モバイルで職務経歴書から飛んでくる採用担当を想定)。パネル padding を詰めて 1 カラムで積む、本体の横スクロールは禁止(図はコンテナ内スクロール)
- `prefers-reduced-motion` で pulse も停止

## 実装メモ

- Next.js `output: 'export'` + GitHub Pages ユーザーページ(`kaionn.github.io`)。**basePath 不要**
- ライブステータスはクライアントコンポーネントで GitHub API を fetch。ISR は使えないので localStorage に短時間キャッシュ(10分程度)。取得失敗時は行ごと非表示
- OGP 画像は後続タスク(初期実装では title/description のメタデータのみ設定)
- 職務経歴書 PDF はリンク先を差し替えられる形に(設定定数化)

## 実装順序

1. セクション枠 + 静的コピー流し込み(ヒーロー〜Contact)
2. プロジェクト2本の本文とパイプライン図
3. ライブステータス(GitHub API 連携)
4. OGP / favicon / reduced-motion などの仕上げ
